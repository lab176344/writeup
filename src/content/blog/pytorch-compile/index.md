PyTorch compile is introduced in PyTorch 2.0, it can improve the performance of existing PyTorch code by JIT compiling it into optimized kernels. JIT trace is the legacy way of doing this, but it has limitations. PyTorch compile addresses these limitations by using a more flexible and powerful approach to compile PyTorch code. It can optimize a wider range of PyTorch code, including dynamic control flow and Python features. However, it is not perfect and there are some cases where it can break. In this blog post, we will take a deep dive into how PyTorch compile works under the hood, and what can break when using it. Compile is a useful tool for the following reasons:

- Requires almost no code changes to existing PyTorch code.
- Will improve the runtime performance with least amount of effort.
- Not platform specific like Tensorrt, which is only for Nvidia GPUs.

## How PyTorch Compile Works

PyTorch compile can be invoked using the `torch.compile` function or the torch.compile decorator.

```python
import torch

torch._logging.set_logs(graph_code=True)

def my_function(x):
    return x * 2

# Option 1: Using torch.compile function

compiled_function = torch.compile(my_function)

# Option 2: Using torch.compile decorator

@torch.compile
def my_function(x):
    return x * 2
```

When the torch.compile method is invoked, PyTorch under the hood performs the following steps:

1. **Step 1**: TorchDynamo Tracing: TorchDynamo is a Python-level JIT compiler that captures the Python bytecode of the function being compiled. It traces the execution of the function and generates an intermediate representation (IR) of the code. This IR is a graph-based representation called FX graph that captures the operations and data flow of the original Python code.

When a torch compile function is called, dynamo

- Installs itself as a frame evaluation function, i.e., it will be called whenever a new Python frame is created. This allows it to capture the execution of the function being compiled.
- Symbolically executes each bytecode instruction of the function.
- Records just the tensor operations and the control flow, dynamo does not trace the tensor values but just the operations on them.
- The trace recording can break in certain cases which we will discuss later in this blog post. When the trace recording breaks, it falls back to executing the original Python code without any optimization.

The FX graph generated by the Dynamo backend is a directed acyclic graph representation of the original Python code. In the graph, the nodes are the ATen operations (aten::add, aten::mul, etc.) and the edges represent the data flow between the operations. Shapes are symbolic with guards, i.e., they are not concrete values but rather placeholders that represent the shape of the tensor. Here guards are conditions that must hold true for the optimized code to be valid. e.g., if the shape of a tensor changes during execution, the guard will fail and the code will fall back to the original Python code.

2. **Step 2**: Graph Lowering with the chosen backend: Graph lowering is the process of lowering the FX graph into optimized kernels that can be executed on the target hardware (CPU, GPU, etc.,). The graph lowering process is performed by the chosen backend. Some of the popular backends are the following,

   - **Inductor**: Inductor is the default backend for PyTorch compile. It is a high-performance code generator that can generate optimized kernels for both CPU and GPU. It uses a combination of techniques such as operator fusion, loop unrolling, and vectorization to optimize the generated code.
   - **TensorRT**: TensorRT is a backend that focuses on optimizing code for Nvidia GPUs. It can generate optimized kernels for Nvidia GPUs, and it is particularly effective for optimizing operations that are common in deep learning workloads.
   - **XLA**: XLA is a backend that focuses on optimizing code for TPUs (Tensor Processing Units). It can generate optimized kernels for TPUs, and it is particularly effective for optimizing operations that are common in deep learning workloads.

The graph lowering process involves several optimization techniques such as operator fusion, loop unrolling, and vectorization. Operator fusion is the process of combining multiple operations into a single kernel to reduce the overhead of launching multiple kernels. For e.g.,

Imperative Code (What You Write):

```python
def forward(self, x):
    a = self.conv1(x)
    b = self.bn1(a)
    c = torch.relu(b)
    return c
```

Traced Graph (What the Compiler Sees):

```text
Input → Conv2d → BatchNorm2d → ReLU → Output
   │        │          │         │
   └────────┴──────────┴─────────┘
     (fused into single kernel)
```

Loop unrolling is the process of unrolling loops to reduce the overhead of loop control. For e.g.,

Imperative Code (What You Write):

```python
for i in range(4):
    x[i] = x[i] * 2
```

Traced Graph (What the Compiler Sees):

```text
x[0] = x[0] * 2
x[1] = x[1] * 2
x[2] = x[2] * 2
x[3] = x[3] * 2
```

Vectorization is the process of using SIMD (Single Instruction, Multiple Data) instructions to perform operations on multiple data elements in parallel.

Imperative Code (What You Write):

```python
for i in range(4):
    x[i] = x[i] * 2
```

Traced Graph (What the Compiler Sees):

```text
x = x * 2  # Vectorized operation using SIMD instructions
```

The graph lowering backend and dynamo has several configurations that can be used to control the optimization process. A full list of these configurations can be found in the [PyTorch documentation](https://pytorch.org/docs/stable/generated/torch.compile.html#torch.compile). As discussed earlier, the graph tracking or creation process can break in certain cases, this is referred to as "graph breaks". In regions of graph breaks, the code will fall back to executing the original Python code without any optimization. An example anatomy of a graph break is shown below

Anatomy of a Graph Break:

```text
Graph Region 1          Graph Break          Graph Region 2
┌────────────────┐     ┌──────────┐        ┌────────────────┐
│  Conv2d        │     │ Python   │        │  Linear        │
│  BatchNorm     │────→│ Fallback │───────→│  Softmax       │
│  ReLU          │     │          │        │                │
└────────────────┘     └──────────┘        └────────────────┘
   (optimized)           (slow!)              (optimized)
```

The Cost of a Graph Break:

1. **Kernel Launch Overhead**: Each graph region requires a separate CUDA kernel launch (~5-20μs)
1. **Memory Round-Trip**: Intermediates must be materialized in GPU memory between regions
1. **Optimization Barrier**: The compiler cannot fuse operations across the break
1. **CPU-GPU Synchronization**: The CPU must coordinate handoffs between regions

With 10 graph breaks, you might spend 1ms just on overhead—larger than the actual computation.

## What Can Break When Using PyTorch Compile

There are several cases where the graph tracking can break during torch compile. Here in this post we will discuss the most common seven cases that can lead to graph breaks.

1. **Data-Dependent Control Flow**: If the control flow of the code depends on the values of tensors, it can lead to graph breaks. The tracer cannot know which branch of the control flow to take during tracing, so it will conservatively assume that both branches can be taken, which can lead to graph breaks. For e.g.,

```python
def my_function(x):
    if x.sum() > 0:
        return x * 2
    else:
        return x * 3
```

```python
def my_function(x):
    if tensor.numel() == 0:
        return x * 2
    else:
        return x * 3
```

All the above examples can lead to graph breaks because the control flow is guarded with tensor data dependent conditions.

2. **Dynamic or Data-dependent Shapes**: If the output shapes of the operations depend on the input data, it can lead to graph breaks. The tracer cannot know the output shapes during tracing, because split values, concatenation dimensions, loop bounds, etc., can depend on the input data. For e.g.,

```python
def my_function(x):
    return x.split(x.shape[0] // 2, dim=0)
```

```python
def my_function(x):
    for i in range(x.shape[0]):
        x[i] = x[i] * 2
    return x
```

3. **CPU-GPU Synchronization**: Any operation that requires the GPU to flush and transfer a value back to the CPU inserts a full device sync. The compiled subgraph before the sync must be emitted and run to completion before execution can continue, forcing a break at that boundary. For e.g.,

```python
x = tensor.item()
sizes = tensor.tolist()
arr = tensor.numpy()
C = cost.cpu()
linear_sum_assignment(C)
```

4. **Unsupported Ops / Dtype Combinations**: If the code contains operations that Dynamo cannot lower to ATen ops, it treats them as opaque calls and breaks the graph. This includes custom CUDA extensions not registered with `torch.library`, certain dtype combinations like bool cumsum, and custom autograd functions without FX support. For e.g.,

```python
bool_tensor.cumsum(0)
ms_deform_attn_cuda(...)
class MyOp(torch.autograd.Function): ...
```

5. **Python-level Callable / Method-pointer Selection**: Even when the condition is a plain Python bool (not a tensor), selecting between two bound method objects produces a Python callable that cannot be represented in the FX graph. Dynamo emits a graph break at the call site where the selected callable is later invoked. This is distinct from data-dependent control flow: the condition itself is not data-dependent; the problem is that the result — a bound method — has no ATen representation. For e.g.,

Here `self.training` is a static bool so the condition is guardable, but the result is a Python callable not an ATen op:

```python
fn = self.segmentation_head.sparse_forward if self.training \
     else self.segmentation_head.forward
fn(x)
```

6. **Observable Side Effects / I/O**: Dynamo must preserve the ordering and observability of side effects relative to tensor operations. It cannot reorder or eliminate a `print` the way it can an elementwise op, so it breaks the graph to let CPython execute the I/O statement and then starts a new compiled region. This is distinct from CPU-GPU synchronization: there is no device sync involved — the break is purely about semantic ordering guarantees for externally-visible effects. For e.g.,

```python
print("loss:", loss.item())
logging.info("step %d", i)
warnings.warn("check input")
```

7. **Unsupported Context Managers Inside a Compiled Region**: `torch.no_grad()` and `torch.autocast` used as context managers inside a compiled function alter gradient-tracking or dtype-dispatch state in ways Dynamo historically could not model within a single FX graph. For e.g.,

```python
@torch.compile
def forward(x):
    with torch.no_grad():
        y = expensive_op(x)
    return y

@torch.no_grad()
def loss_cardinality(outputs, targets):
    ...
```

In modern PyTorch (2.2+) `torch.no_grad()` has partial support inside compiled regions, but `torch.autocast` used as an inner decorator remains a documented break source.

## Performance Benchmark

To see the effect of torch.compile in practice, we benchmark a function that performs 15 sequential elementwise operations on a 2048×2048 tensor on an NVIDIA RTX A5000 GPU. The key here is that each elementwise operation — sin, cos, exp, tanh, sigmoid, etc. — launches a separate CUDA kernel in eager mode. This is exactly the kind of workload where torch.compile's operator fusion should help, because the compiler can collapse all 15 kernel launches into a single fused kernel.

```python
import torch


def my_function(x):
    x = x * 2.0
    x = x + 1.0
    x = torch.sin(x)
    x = x**2
    x = x / 3.0
    x = torch.cos(x)
    x = x - 0.5
    x = torch.exp(x * 0.1)
    x = torch.tanh(x)
    x = x * 1.5
    x = torch.sigmoid(x)
    x = x + torch.sqrt(torch.abs(x) + 1e-6)
    x = torch.relu(x - 0.3)
    x = x**0.5
    x = torch.log(x + 1.0)
    return x


@torch.compile
def my_optimized_function(x):
    x = x * 2.0
    x = x + 1.0
    x = torch.sin(x)
    x = x**2
    x = x / 3.0
    x = torch.cos(x)
    x = x - 0.5
    x = torch.exp(x * 0.1)
    x = torch.tanh(x)
    x = x * 1.5
    x = torch.sigmoid(x)
    x = x + torch.sqrt(torch.abs(x) + 1e-6)
    x = torch.relu(x - 0.3)
    x = x**0.5
    x = torch.log(x + 1.0)
    return x
```

The benchmark runs each function 100 times and compares the steady-state execution time after the first run. The results are shown below.

Unoptimized function (eager mode):

```text
Iteration 0 (first run): 104.220673 ms
Iteration 1:   1.236992 ms
Iteration 2:   1.065984 ms
...
Iteration 10:  1.064960 ms
Iteration 20:  1.065888 ms
...
Iteration 90:  1.064960 ms

Average (after first run): 1.067016 ms
```

Optimized function (torch.compile):

```text
Iteration 0 (compilation + execution): 1414.352905 ms
Iteration 1:   0.178176 ms
Iteration 2:   0.130048 ms
...
Iteration 10:  0.108544 ms
Iteration 20:  0.105472 ms
...
Iteration 90:  0.104448 ms

Average (after compilation): 0.106868 ms
Speedup: 9.98x
```

There are a few things worth noting from the results. The unoptimized function's first run takes 104ms because PyTorch lazily initializes CUDA on the first call — from iteration 2 onwards it stabilizes at around 1.07ms per call. The compiled function's first call takes 1414ms, almost entirely compilation overhead. After that it runs at around 0.107ms per iteration, which is a ~10x speedup over the unoptimized steady-state time.

The reason for this large speedup is operator fusion. In eager mode, each of the 15 operations reads the tensor from GPU memory, performs a small computation, and writes the result back. With fusion, Inductor collapses all 15 operations into a single Triton kernel that reads the input once, performs all the computations in registers, and writes the output once. This eliminates 14 round-trips to GPU memory, which is the bottleneck on a 2048×2048 tensor.

The compilation overhead of ~1.4 seconds means this speedup only pays off if the function is called at least a few hundred times. For a training loop running thousands of iterations, this is an easy tradeoff. For a single inference call, it is not worth it.

## Analysing Graph Breaks

When torch.compile encounters a graph break, it silently falls back to eager execution at that point without raising an error. This makes it easy to miss breaks that are quietly degrading performance. TorchDynamo provides three tools to surface them explicitly.

The first is `torch._logging.set_logs(graph_breaks=True)`, which prints a detailed message every time a break occurs during tracing. The second is `torch._dynamo.explain()`, which runs the tracer over a function and returns a structured summary of every break, the reason for it, and the ops captured in each subgraph between breaks. The third is the `fullgraph=True` option in `torch.compile`, which makes the compiler raise an error immediately if any graph break is detected, instead of silently falling back. This is useful when you want to enforce that your function compiles into a single graph with no fallbacks.

```python
import torch

torch._logging.set_logs(graph_breaks=True)

@torch.compile(fullgraph=True)
def function_no_breaks(x):
    x = x * 2
    x = x + 1
    return torch.relu(x)

@torch.compile
def function_with_breaks(x):
    x = x * 2
    print("intermediate value")
    x = x + 1
    if x.sum() > 0:
        x = torch.relu(x)
    return x

x = torch.randn(4)
function_with_breaks(x)

explanation = torch._dynamo.explain(function_with_breaks)(torch.randn(4))
print(explanation)
```

Running this produces a log message for each break at the point it occurs, showing the reason, a hint for how to fix it, and the exact line in your code that caused it. For the `print` break the message looks like this:

```text
Graph Break Reason: Failed to trace builtin operator
  Explanation: Dynamo does not know how to trace builtin operator `print`
  Hint: Avoid calling builtin `print`. Consider adding it to
        `torch._dynamo.config.reorderable_logging_functions`.
  User code traceback:
    File "graph_data.py", line 11, in function_with_breaks
      print("intermediate value")
```

And for the data-dependent branch:

```text
Graph Break Reason: Data-dependent branching
  Explanation: Detected data-dependent branching (e.g. `if my_tensor.sum() > 0:`).
               Dynamo does not support tracing dynamic control flow.
  Hint: This graph break is fundamental. Consider using `torch.cond` to express
        dynamic control flow.
  User code traceback:
    File "graph_data.py", line 13, in function_with_breaks
      if x.sum() > 0:
```

The `explain()` call at the end prints a summary across all breaks. The key fields to look at are the graph count, break count, and the ops captured per subgraph:

```text
Graph Count: 3
Graph Break Count: 2
Op Count: 4

Break Reasons:
  Break Reason 1: Failed to trace builtin operator (print)
  Break Reason 2: Data-dependent branching (if x.sum() > 0)

Ops per Graph:
  Graph 1: mul
  Graph 2: add, gt
  Graph 3: relu
```

This confirms that the function was split into three subgraphs. Graph 1 captures `x * 2`, Graph 2 captures `x + 1` and the `sum() > 0` comparison, and Graph 3 captures `torch.relu(x)`. The `print` and the branch itself are executed by CPython between the subgraphs.

The `graph_code=True` flag shown in the introductory code example at the top of this post is the complementary tool — it prints the actual generated Triton or C++ kernel code for each compiled subgraph, which is useful when you want to verify that fusion happened as expected.
